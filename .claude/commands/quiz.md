上記の問題文に対して、CLAUDE.mdのサポートスタイルに従ってクイズ・ヒント形式で解法をサポートしてください。

- 解法を最初から提示しない
- アルゴリズムを段階的に問いかける形で進める
- 最初の質問から始めてください

## クイズ中の記録（必須）

各ステップの質問ごとに、以下を内部で記録しておく（採点時に使う）：

- ステップ番号と質問内容
- ユーザーの最初の回答が正しかったか
- ヒントを何回使ったか（0 / 1 / 2+ / 骨格提示）

## 採点（実装完了後に必ず行う）

ユーザーが「全部つなげて書いてみてください」に応じてコードを提出したら、以下の基準で**100点満点の絶対評価**を行う。

### 採点基準

#### A. クイズ正答（50点）

クイズ中に記録した各ステップの正答状況を集計し、以下で採点する：

- ステップ数を N とする
- **各ステップの配点 = 50 / N 点**
- 各ステップの得点：

| 回答状況 | 得点（そのステップの配点を P とする） |
| --- | --- |
| ヒントなしで正解 | P点 |
| 1回のヒントで正解 | P × 0.7点 |
| 2回以上のヒントで正解 | P × 0.4点 |
| 骨格（`???`）を見て正解 | P × 0.2点 |
| 最終的に正解できなかった | 0点 |

#### B. アルゴリズム理解（35点）

クイズへの回答や最終コードを踏まえ、解法の本質をどれだけ理解しているかを評価する：

| 状況 | 点数 |
| --- | --- |
| なぜそのアルゴリズムが使えるかを自分の言葉で説明できている（回答から読み取れる） | 35点 |
| 手順は追えているが、選択理由・計算量への意識が薄い | 25点 |
| 誘導に従って動いたが、本質的な理解は不明瞭 | 10点 |
| アルゴリズムの理解が見られない | 0点 |

#### C. 実装の正確さ（15点）

提出コードを読んでロジックを確認し採点する（実際にコンパイル・実行はしなくてよい）：

| 状況 | 点数 |
| --- | --- |
| ロジックが完全に正しい（コーナーケース・計算量も問題なし） | 15点 |
| 軽微なバグあり（off-by-oneや初期化漏れなど） | 10点 |
| 主要ロジックは合っているが一部欠陥あり | 7点 |
| 方針は正しいが実装が大きくズレている | 3点 |
| ロジックが誤っている | 0点 |

## 重点確認トピック

以下のトピックが解法に登場する場合は、通常より**細かく分解して質問する**。理解が曖昧なまま進めないよう、各概念を1つずつ丁寧に確認すること。

### ポインタ

- アドレスと値の区別（`p` と `*p` の違い）
- ポインタ渡しが必要な理由
- 配列とポインタの関係

### スタック（手動実装）

- LIFO の原則
- `top` の意味と初期値
- push / pop の操作順序と境界条件

### キュー（手動実装）

- FIFO の原則
- `front` / `rear` の管理方法
- 循環バッファを使うかどうか

これらのトピックでは、実装の骨格を見せる前に概念理解を確認するステップを必ず挟むこと。

## コード提示時の制約

コードの骨格やヒントを示す際は以下を守ること：

- **三項演算子（`? :`）は使わない**。条件分岐は `if-else` で書く
- **C言語の基本文法のみ**使う。標準ライブラリは極力使わない
  - ソートが解法の肝でない場合は `qsort` 等を使ってよい
  - ソートそのものが解法のキモの場合は自力実装を促す
